## Finding 1 — Foreign Keys Without ON DELETE CASCADE (AC1, AC4)

**Confidence: 98**

Three foreign key constraints exist without `ON DELETE CASCADE`. When the referenced
parent row is deleted, the database either raises a constraint violation (if FK enforcement
is enabled) or leaves orphaned child rows (if not enabled).

### 1a. `completion_summaries.story_id` — CONFIRMED known issue (AC4)

**`src/main/database/migrations/009_add_completion_summary.js:34`**:
```sql
CREATE TABLE IF NOT EXISTS completion_summaries (
  id TEXT PRIMARY KEY,
  story_id TEXT NOT NULL,
  ...
  FOREIGN KEY (story_id) REFERENCES user_stories(id)
  -- Missing: ON DELETE CASCADE
)
```

**Workaround in place:** Both `archive()` and `delete()` in `user-story-repository.js`
manually delete from `completion_summaries` before deleting the story:

```javascript
// user-story-repository.js:503 (archive)
db.prepare('DELETE FROM completion_summaries WHERE story_id = ?').run(id)

// user-story-repository.js:575 (delete)
db.prepare('DELETE FROM completion_summaries WHERE story_id = ?').run(id)
```

The comments at both locations explicitly acknowledge the constraint:
`// Delete completion_summaries first (FK without CASCADE — must clean up manually)`

**Fragility:** This code-level workaround is the only safeguard. Any deletion path that
does not go through these two methods will hit a FK constraint violation at runtime.
Current deletion paths (`archiveMany()` delegates to `archive()` which is safe), but
future code that deletes stories directly via `db.prepare('DELETE FROM user_stories...')`
would fail.

**Impact:** `FOREIGN KEY constraint failed` error on story delete/archive if any path
is missed. Story becomes undeletable until the orphaned completion_summary is removed.

**Recommended fix:** Add `ON DELETE CASCADE` in a new migration:
```sql
-- New migration: recreate completion_summaries with CASCADE
CREATE TABLE completion_summaries_new (
  id TEXT PRIMARY KEY,
  story_id TEXT NOT NULL,
  ...
  FOREIGN KEY (story_id) REFERENCES user_stories(id) ON DELETE CASCADE
);
INSERT INTO completion_summaries_new SELECT * FROM completion_summaries;
DROP TABLE completion_summaries;
ALTER TABLE completion_summaries_new RENAME TO completion_summaries;
```

### 1b. `ris.plan_id → plans(id)` — no cascade

**`src/main/database/migrations/007_cre_tables.js:42`** (table renamed to `ris` in migration 008):

No cleanup code exists for `ris` records when a plan is deleted. Currently no code path

### 1c. `inspection_assertions.plan_id → plans(id)` — no cascade

**`src/main/database/migrations/007_cre_tables.js:62`** (table renamed in migration 008):

## Finding 2 — DB Writes Without In-Memory Cache Refresh (AC2)

**Confidence: 90**

`PuffinState` uses an in-memory cache (`this.userStories`) alongside SQLite.
Per architecture docs: "Read accessors query SQLite first, update cache from successful
reads, fallback to cache only if query fails. Cache invalidation via `invalidateCache()`
forces next read to query database."

Three write paths update `user_stories.inspection_assertions` directly via raw SQL
without calling `puffinState.invalidateCache(['userStories'])`:

### 2a. CRE assertion generation (`cre/index.js:396`)

```javascript
// cre/index.js:393–398
if (existing) {
  ctx.db.prepare('UPDATE user_stories SET inspection_assertions = ?, updated_at = ? WHERE id = ?')
    .run(JSON.stringify(generated), new Date().toISOString(), storyId)
  // ← No puffinState.invalidateCache() call here
}
```

The CRE module has access to `ctx.db` (raw database connection) but not to `puffinState`.
After this write, if `getUserStories()` is called before the next IPC request flushes
the cache, the renderer receives stale `inspectionAssertions: []`.

**Known symptom:** "Code review modal shows 0/0/0/0 assertions" — documented in CLAUDE.md
gotchas as a recurring bug pattern that required multiple rounds of investigation.

### 2b. Assertion sync in `state:syncAssertionsFromCreTable` (`ipc-handlers.js:682`)

```javascript
// ipc-handlers.js:679–685
db.prepare('UPDATE user_stories SET inspection_assertions = ?, updated_at = ? WHERE id = ?')
  .run(JSON.stringify(assertions), new Date().toISOString(), storyId)
synced++
// ← No invalidateCache() call in the loop or after it
```

The IPC handler `state:syncAssertionsFromCreTable` iterates over story IDs and syncs
assertions from the `inspection_assertions` table to `user_stories`. After the loop
completes, the handler returns `{ success: true, synced }` without invalidating the
puffinState cache.

### 2c. Fallback sync in `getStoryInspectionAssertions()` (`puffin-state.js:687`)

```javascript
// puffin-state.js:685–691
db.prepare('UPDATE user_stories SET inspection_assertions = ?, updated_at = ? WHERE id = ?')
  .run(JSON.stringify(assertions), new Date().toISOString(), storyId)
// ← No this.invalidateCache(['userStories']) call
```

The fallback sync inside `getStoryInspectionAssertions()` writes to the DB but does not
invalidate the cache, so future calls to `getUserStories()` may still return the old
empty `inspectionAssertions` array from the in-memory cache.

**Recommended fix for all three:** After each `UPDATE user_stories SET inspection_assertions`,
call `puffinState.invalidateCache(['userStories'])`. For the CRE module, `ctx` should
expose an `invalidateCache` function — or the post-write invalidation should happen in
the IPC handler that wraps the CRE call.

## Finding 3 — Multi-Step DB Operations Without Transaction (AC3)

**Confidence: 88**

### 3a. CRE dual-table assertion write is not atomic (`cre/index.js:388–400`)

CRE assertion generation writes to two storage locations:

**Step 1** — `inspection_assertions` table (via `assertionGenerator.generate()`, which
calls INSERT internally in `cre-storage.js`).

**Step 2** — `user_stories.inspection_assertions` JSON column:
```javascript
// cre/index.js:396
ctx.db.prepare('UPDATE user_stories SET inspection_assertions = ?, updated_at = ? WHERE id = ?')
  .run(JSON.stringify(generated), new Date().toISOString(), storyId)
```

These two writes are sequential with no wrapping transaction. If step 2 fails (e.g.,
the story was deleted between steps), the `inspection_assertions` table has data that
`user_stories` does not, creating the dual-storage sync divergence that has caused the
`0/0/0/0` assertions bug multiple times.

### 3b. Batch assertion sync has no outer transaction (`ipc-handlers.js:655–690`)

`state:syncAssertionsFromCreTable` iterates over multiple story IDs:
```javascript
// ipc-handlers.js:655–690
for (const storyId of storyIds) {
  const rows = db.prepare('SELECT ... FROM inspection_assertions WHERE story_id = ?').all(storyId)
  if (rows.length > 0) {
    db.prepare('UPDATE user_stories SET inspection_assertions = ? WHERE id = ?')
      .run(JSON.stringify(assertions), ..., storyId)
    synced++
  }
}
```

Each `UPDATE` executes independently. If a crash occurs mid-loop, some stories are
synced and others are not, leaving partial state with no rollback capability.

**Recommended fix:** Wrap both write paths in `db.transaction()`:

```javascript
// For cre/index.js: wrap step 1 + step 2 in a transaction
ctx.db.transaction(() => {
  assertionGenerator.generate(...)  // step 1: inserts to inspection_assertions
  ctx.db.prepare('UPDATE user_stories SET inspection_assertions = ?...').run(...)  // step 2
})()

// For ipc-handlers.js: wrap the loop
db.transaction(() => {
  for (const storyId of storyIds) {
    // ... existing per-story logic
  }
})()
```

## Finding 4 — Unique Constraint on `inspection_assertions.id` (AC5)

**Confidence: 95** | **Result: COMPLIANT (fix in place)**

`inspection_assertions` has `id TEXT PRIMARY KEY`, which SQLite enforces as a UNIQUE
NOT NULL constraint.

**Historical violation (now fixed):** The CRE-generated assertion IDs used sequential
strings (`IA001`, `IA002`, ...) that were reused across stories. When story B's
assertions were inserted after story A's, the duplicate `IA001` hit the PRIMARY KEY
constraint, causing the entire INSERT to fail silently — resulting in 0 assertions stored.

**Current state:** `src/main/cre/assertion-generator.js:240` always calls `uuidv4()`,
ignoring any AI-provided IDs:
```javascript
_validateAssertion(assertion) {
  return {
    id: uuidv4(), // Always generate a unique ID — AI-generated IDs (e.g. "IA001") collide
    type,
    // ...
  }
}
```

This fix is correctly in place. The UUID approach guarantees no collisions regardless
of AI output.

## Finding 5 — `plans.sprint_id` Has Index but No FK Constraint (Bonus)

**Confidence: 85** | **Severity: Low**

**`src/main/database/migrations/008_cre_schema_rename.js:48`**:
```sql
CREATE UNIQUE INDEX idx_plans_sprint_id ON plans(sprint_id)
```

`plans.sprint_id` has a UNIQUE index but no `FOREIGN KEY (sprint_id) REFERENCES sprints(id)`
constraint. This means:
- A plan can reference a sprint ID that does not exist in the `sprints` table.
- Deleting a sprint does not cascade to remove its associated plan.
- The uniqueness constraint (one plan per sprint) is enforced, but referential integrity is not.

**Risk:** Low today — plans are created alongside sprints in the same operation.
Future code that deletes sprints independently could leave orphaned plans.

**Recommended fix:** Add a FK constraint in a new migration. Since migration 008 used
`ALTER TABLE`, and SQLite does not support adding FKs via ALTER TABLE, the table would
need recreation. Consider deferring this until another plan-schema migration is needed.

## Finding 6 — `archive()` Uses Deferred `transaction()` for 3-Step Write (Bonus)

**Confidence: 80** | **Severity: Low**

**`src/main/database/repositories/user-story-repository.js:466`**:
```javascript
archive(id) {
  return this.transaction(() => {        // ← deferred transaction
    db.prepare(insertSql).run(...)        // step 1: INSERT into archived_stories
    db.prepare('DELETE FROM completion_summaries WHERE story_id = ?').run(id)  // step 2
    db.prepare('DELETE FROM user_stories WHERE id = ?').run(id)  // step 3
  })
}
```

`this.transaction()` uses `db.transaction(fn)` — a **deferred** transaction (starts as
a read lock, upgrades to write lock on first write). Under concurrent write load, this
can cause `SQLITE_BUSY` if another writer holds a write lock at the moment of upgrade.

`delete()` at line 550 correctly uses `this.immediateTransaction()` which acquires a
write lock immediately. For symmetry and robustness, `archive()` should use the same
approach given that all three steps are writes.

**Note:** `archiveMany()` (line 522) calls `archive()` from within its own
`this.transaction()` — better-sqlite3 handles nested `db.transaction()` calls via
savepoints automatically, so there is no nested-transaction issue.

**Recommended fix:** Change line 466 from `this.transaction(` to
`this.immediateTransaction(`. This is a one-word change.

## Prioritized Recommendations

### High Priority

1. **Add migration to drop and recreate `completion_summaries` with `ON DELETE CASCADE`**
   (Finding 1a / AC4): Eliminates the fragile code-level workaround. Any missed deletion
   path currently risks a runtime FK constraint violation.

2. **Add `invalidateCache(['userStories'])` after all three bare `UPDATE user_stories SET inspection_assertions` calls** (Finding 2): Prevents stale assertion data in the renderer.
   Locations: `cre/index.js:397`, `ipc-handlers.js:683`, `puffin-state.js:688`.

3. **Wrap the dual-table CRE assertion write in a transaction** (Finding 3a):
   Makes the `inspection_assertions` INSERT and `user_stories` UPDATE atomic, eliminating
   the root cause of the dual-storage divergence that has caused the 0/0/0/0 bug.

### Medium Priority

4. **Wrap `state:syncAssertionsFromCreTable` loop in a transaction** (Finding 3b): Ensures
   batch assertion syncs are atomic and can be rolled back on partial failure.

5. **Add `ON DELETE CASCADE` to `ris.plan_id` and `inspection_assertions.plan_id`**
   (Findings 1b, 1c): Low urgency (no current deletion path for plans), but necessary
   before any plan management UI is added.

### Low Priority

6. **Change `archive()` to use `immediateTransaction()`** (Finding 6): One-word change
   for consistency and resilience under write contention.

7. **Add FK constraint for `plans.sprint_id → sprints(id)`** (Finding 5): Deferred until
   next plan-schema migration since SQLite requires table recreation to add FKs.
