## Finding 3 — Response Envelope Missing `success` Key

**Confidence: 95**

**Convention:** All IPC handlers must return `{ success: boolean, data?, error? }` so callers can gate on `result.success` before accessing payload fields.

**Violations:** Four handlers return custom shapes without a `success` key.

### 3a. `app:getRecentProjects` — returns raw array

```javascript
// src/main/main.js:441
ipcMain.handle('app:getRecentProjects', () => {
  return getRecentProjects()  // returns string[] — no success wrapper
})
```

**Expected:**
```javascript
return { success: true, projects: getRecentProjects() }
```

### 3b. `tools:checkSnip` — returns `{ installed }` without `success`

```javascript
// src/main/ipc-handlers.js:253–255
return { installed: true, path: result }
// OR
return { installed: false }
```

**Expected:**
```javascript
return { success: true, installed: true, path: result }
return { success: true, installed: false }
```

### 3c. `claude:check` — returns `{ available, version }` without `success`

```javascript
// src/main/ipc-handlers.js:1514–1517
return { available, version }
// OR (in catch)
return { available: false, version: null }
```

**Expected:**
```javascript
return { success: true, available, version }
```

### 3d. `claude:isRunning` — returns a raw boolean

```javascript
// src/main/ipc-handlers.js:1524
return claudeService.isProcessRunning()  // boolean
```

**Expected:**
```javascript
return { success: true, running: claudeService.isProcessRunning() }
```

**Risk:** Renderer callers accessing `result.success` on these responses will get `undefined` (falsy), potentially treating a successful response as an error if caller code is generalized.

## Finding 4 — Plugin and Scheduler Handlers Registered After `createWindow()`

**Confidence: 80**

**Convention:** Per `CLAUDE.md`: "All IPC handlers … are registered in `setupIpcHandlers()` which is called BEFORE `createWindow()` in main.js."

**Violation:** Five IPC setup functions are called inside `initializeProject()` (`src/main/main.js:342–380`), which always executes after `createWindow()` returns.

| Function | Called at line | Registers channels |
|---|---|---|
| `setupPluginHandlers()` | 342 | `plugins:list`, `plugins:get`, `plugins:listLoaded`, `plugins:listFailed`, `plugins:getErrors`, `plugins:getSummary`, `plugins:reload`, `plugins:getDirectory`, `plugins:enable`, `plugins:disable`, `plugins:listActive`, `plugins:getState`, `plugins:getInfo`, `plugins:getFullSummary`, `plugins:reloadPlugin`, `plugins:getRegistrySummary`, `plugins:getActions`, `plugins:getComponents` |
| `setupPluginManagerHandlers()` | 376 | `plugins:*` manager channels |
| `setupViewRegistryHandlers()` | 377 | `plugin:register-view`, `plugin:unregister-view`, `plugin:get-sidebar-views`, `plugin:get-views-by-location`, `plugin:get-all-views`, `plugin:get-plugin-views`, `plugin:get-view`, `plugin:get-view-summary` |
| `setupPluginStyleHandlers()` | 378 | `plugin:get-style-paths`, `plugin:get-all-style-paths`, `plugin:get-renderer-config` |
| `setupSchedulerHandlers()` | 380 | `schedule:list`, `schedule:create`, `schedule:update`, `schedule:delete`, `schedule:runNow` |

**Context:** These handlers require project-specific state (plugin loader, plugin manager, scheduler instance) that is not available until a project is opened. The architecture intentionally defers registration. However, if the renderer calls any of these channels before the project is loaded (e.g., from a plugin panel rendered eagerly), Electron will throw `Error: No handler registered for 'plugins:list'`.

**Mitigation already in place for CRE:** `cre.registerHandlers()` is called from `setupIpcHandlers()` (line 115) with an `initialized` guard inside each handler. This pattern should be applied to plugin and scheduler handlers.

## Finding 5 — `ipcMain.on` Usage Review (No Violations)

**Confidence: 90**

Four channels use `ipcMain.on` instead of `ipcMain.handle`:

| Channel | Line | Rationale |
|---|---|---|
| `claude:submit` | 1532 | Streaming — sends multiple `claude:*` events back; single return value is insufficient |
| `claude:cancel` | 1665 | True fire-and-forget — no return value expected |
| `claude:deriveStories` | 1705 | Streaming — sends `claude:storyDerivationError` / story events back |
| `claude:modifyStories` | 1768 | Streaming — sends modification events back |

All four are correct. Streaming operations cannot use `ipcMain.handle` (which only allows one return value). No misuse detected.
