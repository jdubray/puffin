## Finding 1 — Event Listeners Without Cleanup (AC1)

**Confidence: 92**

The preload bridge is well-designed: every `on*` method wraps `ipcRenderer.on` and
returns an unsubscribe function: `() => ipcRenderer.removeListener(channel, handler)`.
The `claudeListeners` array collects unsubscribe functions for all streaming session
listeners and `destroy()` at `app.js:7191` calls them all correctly.

Four categories of registrations bypass this pattern:

### 1a. Menu profile handlers registered twice (12 listeners for 6 channels)

`setupProfileMenuHandlers()` at `app.js:1915–1924` registers 6 handlers:
```javascript
window.puffin.menu.onProfileView(() => this.handleProfileAction('view'))
window.puffin.menu.onProfileCreate(() => this.handleProfileAction('create'))
window.puffin.menu.onProfileEdit(() => this.handleProfileAction('edit'))
window.puffin.menu.onProfileExport(() => this.handleProfileAction('export'))
window.puffin.menu.onProfileImport(() => this.handleProfileAction('import'))
window.puffin.menu.onProfileDelete(() => this.handleProfileAction('delete'))
```

`setupMenuListeners()` at `app.js:2056–2103` independently registers the same 6
channels with different callbacks:
```javascript
window.puffin.menu.onProfileView(() => { this.intents.showModal('profile-view', {}) })
window.puffin.menu.onProfileCreate(() => { this.intents.showModal('profile-create', {}) })
// ... etc for all 6
```

Both methods are called during initialization. Neither stores the returned unsubscribe
functions, and neither set is tracked in `claudeListeners`. Result: 12 `ipcRenderer.on`
registrations on 6 channels, and every `profile:view` (etc.) event dispatches two
handlers — one that calls `handleProfileAction()` and one that shows the modal directly.

**Impact:**
- Duplicate IPC listener overhead on every menu event.
- Neither listener is ever removed, so this leaks for the application lifetime.
- Both handlers fire simultaneously on the same event, which can cause double modal
  openings or double `handleProfileAction` calls depending on call order.

**Fix:** Consolidate to one registration site and store the returned unsub functions
in `claudeListeners`. Remove `setupProfileMenuHandlers()` and keep only
`setupMenuListeners()` (or vice versa), ensuring the return values are tracked:
```javascript
this.claudeListeners.push(window.puffin.menu.onProfileView(() => { ... }))
this.claudeListeners.push(window.puffin.menu.onProfileCreate(() => { ... }))
// ... etc
```

### 1b. Assertion evaluation listeners registered inside render-path code

At `app.js:3294–3312`, inside a function that re-renders the sprint context panel:
```javascript
if (window.puffin?.state?.onAssertionEvaluationComplete) {
  window.puffin.state.onAssertionEvaluationComplete((data) => {    // ← line 3295
    this.intents.updateStoryAssertionResults(data.storyId, data.results)
  })
}
if (window.puffin?.state?.onAssertionEvaluationProgress) {
  window.puffin.state.onAssertionEvaluationProgress((data) => {    // ← line 3306
    const section = sprintContextPanel.querySelector(...)
  })
}
```

This code runs every time the sprint panel is updated (which is driven by the SAM
render cycle — potentially many times per sprint session). Each invocation adds a new
`ipcRenderer.on` listener for `assertion-evaluation-complete` and
`assertion-evaluation-progress`. The returned unsubscribe functions are not stored.

**Impact:** N listeners accumulate for each re-render of the sprint panel. Each fires
on every evaluation event. After a busy sprint session with frequent re-renders, dozens
of handlers can fire for a single assertion evaluation event.

**Fix:** Register these listeners once at startup (alongside the other global listeners
in `initSam()`) and add the unsub functions to `claudeListeners`:
```javascript
this.claudeListeners.push(
  window.puffin.state.onAssertionEvaluationComplete((data) => {
    this.intents.updateStoryAssertionResults(data.storyId, data.results)
  })
)
this.claudeListeners.push(
  window.puffin.state.onAssertionEvaluationProgress((data) => {
    // update progress in whatever panel is currently mounted
  })
)
```

### 1c. Puppeteer listeners use raw `ipcRenderer.on` (no removable reference)

At `app.js:4412–4421`:
```javascript
window.puffin.puppeteer.onScreenshot(({ count }) => {    // ← line 4413
  this._updateScreenshotBadge(count)
})
window.puffin.puppeteer.onVerdict(({ verdict }) => {     // ← line 4418
  this._updatePuppeteerVerdict(verdict)
})
```

The preload bridge for puppeteer (`preload.js:922–924`) uses raw `ipcRenderer.on`
without returning a cleanup function:
```javascript
onScreenshot: (cb) => ipcRenderer.on('claude:puppeteer-screenshot', (e, data) => cb(data)),
onVerdict:    (cb) => ipcRenderer.on('claude:puppeteer-verdict',    (e, data) => cb(data))
```

Unlike all other preload `on*` methods (which wrap and return `removeListener`),
`onScreenshot` and `onVerdict` return the `EventEmitter` object — not a cleanup
callback. Neither call site stores or calls any cleanup.

**Impact:** If the puppeteer panel is initialized more than once (e.g., after the UI
re-initializes), listeners accumulate on both channels. The `cb` closures capture
`this` (the app instance), which also prevents garbage collection of the app object
while these listeners remain registered.

**Fix (two-part):**
1. Update preload to return an unsubscribe function (consistent with all other channels):
```javascript
onScreenshot: (cb) => {
  const h = (e, data) => cb(data)
  ipcRenderer.on('claude:puppeteer-screenshot', h)
  return () => ipcRenderer.removeListener('claude:puppeteer-screenshot', h)
},
onVerdict: (cb) => {
  const h = (e, data) => cb(data)
  ipcRenderer.on('claude:puppeteer-verdict', h)
  return () => ipcRenderer.removeListener('claude:puppeteer-verdict', h)
}
```
2. Store the return values in `claudeListeners` at the call sites.

### 1d. Plugin and app-ready listeners not tracked

Four global listeners registered at init time store unsub functions in local scope
but not in `claudeListeners`:

| Location | Channel | Unsub stored? |
|---|---|---|
| `app.js:942` | `app:ready` | No — return value discarded |
| `app.js:955` | `app:projectReady` | No — return value discarded |
| `app.js:1265` | `plugin:activated` | No — return value discarded |
| `app.js:1278` | `plugin:deactivated` | No — return value discarded |

**Impact:** These are application-lifetime listeners and the app itself is never
destroyed while running, so practical leakage is nil. However, they prevent these
channels from being cleaned up if `destroy()` is called during testing or hot-reload
scenarios. A process restart also clears them (Electron restarts the renderer).

**Risk: Low** — Document the intentional lifetime. Alternatively, push to `claudeListeners`
for correctness.

## Finding 3 — MetricsService Singleton Cleanup (AC3)

**Confidence: 97** | **Result: COMPLIANT**

`src/main/main.js:528` calls `shutdownMetricsService()` inside the `before-quit` handler:
```javascript
app.on('before-quit', async (event) => {
  await shutdownMetricsService()   // line 528 — flushes pending batch, stops timer
  await cre.shutdown()             // line 534 — releases process lock
  // ...
  await pluginManager.shutdown()   // line 540 — deactivates plugins
})
```

`shutdownMetricsService()` (`metrics-service.js:616`) flushes the batch buffer,
calls `clearInterval(this.flushTimer)` (line 154), and sets the module-level singleton
to `null` (preventing future `getMetricsService()` calls from returning a stale instance).

One nuance: if `initializeMetricsService()` is called a second time (in the same process),
the existing singleton would be reused without reinitializing the timer. The
`state:init` handler calls `initializeMetricsService(db)` once, and `initializeProject()`
is guarded by `projectInitialized = true` preventing double init. **No issue.**

## Finding 4 — Circular References (AC4)

**Confidence: 90** | **Result: COMPLIANT**

Searched all renderer source for `this.app =`, `this.parent =`, `this.owner =` patterns
— no matches found. Components do not store back-references to the `PuffinApp` instance.

Component reference graph flows in one direction:
```
PuffinApp.components{}
  └── UserStoriesComponent
  └── ModalManager
  └── StatePersistence
  └── ... (all others)
```

`destroy()` at `app.js:7194` iterates `Object.values(this.components)` and calls each
component's `destroy()` method. Components call into `PuffinApp` only via the `intents`
object (passed at construction time), not via a stored reference to the instance.

The `intents` object itself is created by the SAM framework and holds action functions —
it does not hold a reference back to `PuffinApp`. **No circular references detected.**

## Finding 5 — Cached Data Never Invalidated (AC5)

**Confidence: 85** | **Result: Low risk**

### 5a. `_claudeIsExe` — process-lifetime binary-type cache

`src/main/claude-service.js:1235–1246` caches whether the `claude` binary is a
native `.exe` vs a `.cmd` wrapper:
```javascript
if (this._claudeIsExe === undefined) {
  const claudePath = execSync('where claude', ...).trim().split('\n')[0].trim()
  this._claudeIsExe = claudePath.endsWith('.exe')
}
```

This value is computed once and never cleared. If a user upgrades Claude from a `.cmd`
wrapper version to a native `.exe` (or vice versa) while Puffin is running, the
spawn behavior would be stale until restart.

**Risk: Very low** — Claude binary type does not change during normal development
sessions. The cache is appropriate for the process lifetime.

### 5b. `puffinState` in-memory cache — well-managed

The main-process state cache (`puffin-state.js`) uses `invalidateCache(['userStories'])`,
`invalidateCache(['activeSprint'])`, etc. at 17 locations. The pattern correctly follows:
*write to DB → invalidate cache → next read re-queries DB*.

One gap already documented in the database review: three `UPDATE user_stories SET
inspection_assertions` calls (in `cre/index.js:396`, `ipc-handlers.js:682`, and
`puffin-state.js:687`) do not call `invalidateCache()` after writing. This means
the `userStories` in-memory cache can serve stale assertion data until the next
explicit invalidation. See database integrity review for full details.

### 5c. Plugin registry — no persistent cache concern

The plugin registry (`plugin-manager.js`) maintains an in-memory map of active plugins,
cleared fully on `shutdown()`. Loaded plugin modules are held in Node's module cache
(`require` cache), which persists for the process lifetime — by design.
