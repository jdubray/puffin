## Finding 1 — Direct State Mutations (AC1)

**Confidence: 95** | **Result: COMPLIANT**

Searched for `this\.state\.\w+ =` assignments in `src/renderer/app.js` — no matches found.

All `this.state.*` accesses in the render callback are read-only. State is modified exclusively through acceptors in `src/renderer/sam/model.js`, which receive `model` (the live object) and `proposal` (the dispatched action). This is the correct SAM mutation boundary.

**One borderline case noted (not a violation):**

`src/renderer/lib/state-persistence.js:239` mutates the transient `state` copy:

```javascript
state.userStories[index].inspectionAssertions = result.story.inspectionAssertions
```

This is inside the persistence layer's `persist()` method, which receives the transient state snapshot as its argument. The mutation updates the in-memory copy with freshly server-generated assertions after an `ADD_USER_STORY` IPC call completes. Since `persist()` operates on the transient state (not the live model), this does not violate the SAM pattern. However, the SAM-correct alternative would be dispatching `updateUserStory` to let the acceptor update the model.

## Finding 2 — persistActions Whitelist vs Handler Condition Block Parity (AC2)

**Confidence: 93** | **Result: VIOLATIONS**

`state-persistence.js` enforces a two-step gate:

1. **Whitelist gate** (line 138): `if (!persistActions.includes(normalizedType)) return` — actions not in the whitelist are silently dropped before any persistence code runs.
2. **Handler condition blocks** (lines 145–560): each block checks `normalizedType` to choose the right IPC call.

**The two sets must match exactly.** Violations in either direction cause silent failures:
- Action in whitelist but not in any handler → passes gate, nothing persists (silent no-op).
- Action in handler but not in whitelist → handler code is dead (whitelist gate returns first).

### 2b — 1 action type in whitelist but not in any handler block (silent no-op persist)

`ITERATE_SPRINT_PLAN` appears in the whitelist at line 122 but does not appear in any handler condition block. When dispatched:
- It passes the whitelist gate.
- No condition block matches it.
- `persist()` runs without calling any IPC — a silent no-op.

**Impact:** Iterating the sprint plan (refining CRE-generated plans) does not persist the updated plan to SQLite. Restarting loses the iteration.

**Fix:** Add `'ITERATE_SPRINT_PLAN'` to the sprint handler condition block alongside `'SET_SPRINT_PLAN'` (line 268). Both should call `updateActiveSprint`.

## Finding 4 — Pending Flags Not Cleared on Error Paths (AC4)

**Confidence: 88** | **Result: VIOLATION**

`src/renderer/sam/model.js` has two acceptors relevant to CRE workflow error handling:

**`crePlanningCompleteAcceptor`** (success path, line ~2099):
```javascript
model._pendingCrePlanning = null    // ✓ cleared
model._pendingCreIteration = null   // ✓ cleared
model._pendingCreAnswers = null     // ✓ cleared
model._pendingCreApproval = null    // ✓ cleared
```

**`crePlanningErrorAcceptor`** (error path, line ~2109):
```javascript
model.activeSprint.status = 'created'
model._pendingCrePlanning = null    // ✓ cleared
model._pendingCreIteration = null   // ✓ cleared
model._pendingCreAnswers = null     // ✓ cleared
// ✗ model._pendingCreApproval is NOT cleared
```

`_pendingCreApproval` is only cleared on success (`crePlanningCompleteAcceptor`) and on app initialization (`loadStateAcceptor`, line 306). It is **not cleared** in `crePlanningErrorAcceptor`.

**Scenario where this causes a bug:**

1. User clicks "Approve Plan" → `_pendingCreApproval` is set.
2. `handleCreApproval` fires. It internally triggers `handleCrePlanning` as step 1.
3. CRE planning fails → `CRE_PLANNING_ERROR` is dispatched → `crePlanningErrorAcceptor` runs.
4. `_pendingCreApproval` remains non-null.
5. On next SAM render pass, `if (state._pendingCreApproval && !this._handlingCreApproval)` fires again.
6. `handleCreApproval` re-enters immediately, before the user has acknowledged the error.

**Fix:**

```javascript
// crePlanningErrorAcceptor in model.js
export const crePlanningErrorAcceptor = model => proposal => {
  if (proposal?.type === 'CRE_PLANNING_ERROR') {
    if (!model.activeSprint) return

    model.activeSprint.status = 'created'
    model._pendingCrePlanning = null
    model._pendingCreIteration = null
    model._pendingCreAnswers = null
    model._pendingCreApproval = null   // ← add this line
    model.sprintError = { ... }
  }
}
```

**Additional note:** `CLEAR_PENDING_CRE_APPROVAL` action and its acceptor are absent. The only mechanism to clear `_pendingCreApproval` outside of success is app restart (via `loadStateAcceptor`). Adding a dedicated clear action would follow the same pattern as `CLEAR_PENDING_CRE_PLANNING`, `CLEAR_PENDING_CRE_ANSWERS`, etc.

## Recommendations

### High Priority

1. **Add 17 missing action types to `persistActions` whitelist** (Finding 2a): Code review, bug fix, and orchestration state changes are silently lost on restart. This is a data loss bug affecting all users who use sprint orchestration or code review.

2. **Add `ITERATE_SPRINT_PLAN` to the sprint handler condition block** (Finding 2b): Sprint plan iterations silently fail to persist.

### Medium Priority

3. **Clear `_pendingCreApproval` in `crePlanningErrorAcceptor`** (Finding 4): One-line fix prevents re-entry of the approval handler after a planning error. Consider also adding a dedicated `CLEAR_PENDING_CRE_APPROVAL` action for symmetry with other pending-clear actions.

### Low Priority

4. **State-persistence transient mutation in `ADD_USER_STORY` handler** (Finding 1 borderline): `state.userStories[index].inspectionAssertions = ...` mutates the transient SAM state snapshot. Consider dispatching `this.intents.updateUserStory(id, { inspectionAssertions: ... })` instead, so the change flows through the acceptor and reaches the live model cleanly.
