## Finding 1 — innerHTML with Unsanitized Markdown (AC1)

**Confidence: 92** | **Severity: High**

### 1a. `renderMarkdown()` does not escape input before HTML injection

`src/renderer/app.js:5015–5047` — the `renderMarkdown()` method applies regex-based
markdown-to-HTML conversion directly on the raw input text without first escaping HTML
special characters:

```javascript
renderMarkdown(text) {
  if (!text) return ''
  return text
    .replace(/^### (.+)$/gm, '<h4>$1</h4>')  // $1 = unescaped user/AI text
    .replace(/^## (.+)$/gm,  '<h3>$1</h3>')
    .replace(/^# (.+)$/gm,   '<h2>$1</h2>')
    .replace(/\*\*(.+?)\*\*/g, '<strong>$1</strong>')
    .replace(/`([^`]+)`/g,    '<code>$1</code>')
    // ...
}
```

If `text` contains `<img src=x onerror="alert(1)">`, the raw tag passes through all
`replace()` calls unchanged and lands in the DOM. Similarly, `<script>` and event
handler attributes survive because no replacement rule removes arbitrary HTML.

### 1b. Two call sites inject `renderMarkdown()` output directly into `innerHTML`

**`src/renderer/app.js:1882`** — renders a handoff summary restored from `localStorage`:
```javascript
summaryDisplay.innerHTML = `<div class="handoff-summary-content">${this.renderMarkdown(data.summary)}</div>`
```
`data.summary` is stored by `localStorage.setItem(...)` (line 4899), populated from
`response.response` — the raw text response from a Claude CLI invocation. Claude output
is AI-generated but can be influenced by adversarial content in prompts or project files.

**`src/renderer/app.js:4902`** — same assignment at display time:
```javascript
summaryDisplay.innerHTML = `<div class="handoff-summary-content">${this.renderMarkdown(response.response)}</div>`
```

**Risk model:** The renderer runs in a sandboxed BrowserWindow with `nodeIntegration:false`
and `contextIsolation:true`. A successful XSS cannot reach Node.js APIs directly.
However, it can exfiltrate text visible in the renderer (project paths, story content,
sprint data) by constructing a `fetch()` or `XMLHttpRequest` to an attacker-controlled
server if CSP does not block outbound requests.

**Fix:** Escape HTML before running regex replacements:
```javascript
renderMarkdown(text) {
  if (!text) return ''
  const escaped = text
    .replace(/&/g, '&amp;').replace(/</g, '&lt;')
    .replace(/>/g, '&gt;').replace(/"/g, '&quot;')
  return escaped
    .replace(/^### (.+)$/gm, '<h4>$1</h4>')
    // ... rest of replacements unchanged
}
```
This ensures `<img onerror=...>` becomes `&lt;img onerror=...&gt;` before any rule runs.

### 1c. Error message injected without escaping in `user-stories.js`

**`src/renderer/components/user-stories/user-stories.js:1826`**:
```javascript
summaryEl.innerHTML = `<span class="assertion-icon error">!</span> Error: ${errorMessage}`
```

`errorMessage` comes from IPC error responses (e.g., assertion evaluation failures).
While IPC error strings originate from the main process (not direct user input), they
can contain file paths or database values that embed special characters. An assertion
with a title containing `<b>` would produce broken HTML; a deliberately crafted entry
could inject attributes or tags.

**Fix:**
```javascript
summaryEl.innerHTML = `<span class="assertion-icon error">!</span> Error: ${this.escapeHtml(errorMessage)}`
```
`escapeHtml()` already exists in the component's parent class at `app.js:223`.

## Finding 3 — Process Spawning with `shell: true` (AC3)

**Confidence: 88** | **Severity: Low**

### 3a. `taskkill` via `exec()` with numeric PID interpolation

**`src/main/claude-service.js:129`**:
```javascript
exec(`taskkill /pid ${proc.pid} /T /F`, (err) => { ... })
```

`proc.pid` is an integer property of a Node.js `ChildProcess` object created internally
— it cannot be controlled by renderer IPC input. Integer PIDs cannot be used for shell
injection. **No vulnerability.**

### 3b. `shell: true` usage for `.cmd` wrapper detection

**`src/main/claude-service.js:1206–1246`** — `getSpawnOptions()` sets `shell: true`
only when `claude` resolves to a `.cmd` file (not a native `.exe`). When `shell: true`,
all CLI arguments are built as an array passed to `spawn()`, never via string template
interpolation. The `where claude` resolution itself runs `execSync('where claude', ...)`
with no user-provided input. **No vulnerability.**

### 3c. `execSync` for `snip` availability check

**`src/main/ipc-handlers.js:252`**:
```javascript
const cmd = process.platform === 'win32' ? 'where snip' : 'which snip'
const result = execSync(cmd, { encoding: 'utf8', stdio: ['pipe', 'pipe', 'pipe'] }).trim()
```

Static string — no interpolation. **No vulnerability.**

## Finding 4 — IPC Inputs Not Validated Before Use (AC4)

**Confidence: 90** | **Severity: Medium**

### 4a. `shell:openExternal` accepts any URL protocol

**`src/main/ipc-handlers.js:2560–2567`**:
```javascript
ipcMain.handle('shell:openExternal', async (event, url) => {
  try {
    await shell.openExternal(url)   // url is passed verbatim
    return { success: true }
  } catch (error) {
    return { success: false, error: error.message }
  }
})
```

No protocol validation is applied. Electron's `shell.openExternal()` will attempt to
open any URL including `file:///`, `smb://`, `ftp://`, or application-protocol URLs
(`ms-calculator://`, etc.). On Windows, certain protocol handlers can be used to execute
programs.

The same pattern appears at **`src/main/ipc-handlers.js:3373`** (`webserver:openUrl`):
```javascript
ipcMain.handle('webserver:openUrl', async (event, url) => {
  await shell.openExternal(url)
})
```

**Fix:** Add protocol allowlisting before calling `openExternal`:
```javascript
ipcMain.handle('shell:openExternal', async (event, url) => {
  try {
    const parsed = new URL(url)
    if (!['http:', 'https:'].includes(parsed.protocol)) {
      return { success: false, error: 'Only http/https URLs are allowed' }
    }
    await shell.openExternal(url)
    return { success: true }
  } catch (error) {
    return { success: false, error: error.message }
  }
})
```

### 4b. `markdown:parse` accepts unvalidated renderer options

**`src/main/ipc-handlers.js:2570–2577`**:
```javascript
ipcMain.handle('markdown:parse', async (event, content, options = {}) => {
  try {
    const html = marked.parse(content, options)
    return { success: true, html }
  } catch (error) {
    return { success: false, error: error.message }
  }
})
```

The `options` object is forwarded directly to `marked.parse()`. The `marked` library
accepts a `renderer` property in options that allows overriding how HTML tags are
constructed. A renderer that bypasses escaping or adds script injection could be
supplied by the renderer process. While the renderer is not directly attacker-controlled,
this creates a defence-in-depth gap.

**Fix:** Whitelist safe `marked` options and ignore the rest:
```javascript
ipcMain.handle('markdown:parse', async (event, content, options = {}) => {
  try {
    const safeOptions = { gfm: options.gfm ?? true, breaks: options.breaks ?? false }
    const html = marked.parse(content, safeOptions)
    return { success: true, html }
  } catch (error) {
    return { success: false, error: error.message }
  }
})
```

## Finding 5 — Tools Enabled for Untrusted AI Content (AC5)

**Confidence: 95** | **Result: COMPLIANT**

All CRE prompt builders correctly set `disableTools: true` for one-shot invocations:

| File | Line | Flag |
|---|---|---|
| `src/main/cre/assertion-generator.js` | 161 | `disableTools: true` |
| `src/main/cre/plan-generator.js` | 183, 262, 360 | `disableTools: true` |

`disableTools: true` creates an empty MCP config file (`puffin-empty-mcp.json`) and
passes `--strict-mcp-config` to the Claude CLI, preventing the model from receiving
`hdsl_*` tool definitions. This stops the AI from spending turns on codebase exploration
instead of producing structured JSON output — and prevents tool-use responses from
reaching code that expects plain JSON.

`includeToolGuidance: false` is also set in all prompt builders that call `sendCrePrompt()`,
removing the `CODE_MODEL_TOOLS_BLOCK` instruction from the prompt text itself.

## Finding 6 — SQL Injection via Unvalidated Column Interpolation (Latent Risk)

**Confidence: 82** | **Severity: Medium (currently unexploited; architectural risk)**

**`src/main/database/repositories/user-story-repository.js:187–194`**:
```javascript
findAll(options = {}) {
  const { orderBy = 'created_at', order = 'DESC', limit, offset } = options

  let sql = `SELECT * FROM user_stories WHERE status != 'archived' ORDER BY ${orderBy} ${order}`

  if (limit) {
    sql += ` LIMIT ${limit}`
    if (offset) {
      sql += ` OFFSET ${offset}`
    }
  }

  return db.prepare(sql).all()
}
```

Four values are interpolated directly into SQL: `orderBy`, `order`, `limit`, `offset`.
SQLite's `db.prepare()` parameterizes values with `?` placeholders, but **column names
and SQL keywords cannot be parameterized** — they must be validated at the application
layer before interpolation.

**Current exploitation path:** All callers (`puffin-state.js:201`, `puffin-state.js:623`,
`puffin-state.js:2445`, `sprint-scheduler-service.js:235`) pass no `options` argument,
using the safe defaults. **No IPC handler currently exposes `options` to the renderer.**
The risk is latent — a future IPC handler that forwards renderer-supplied sort/page
parameters would open a SQL injection vector.

**Fix:** Validate before interpolation:
```javascript
findAll(options = {}) {
  const ALLOWED_ORDER_BY = new Set(['created_at', 'updated_at', 'title', 'status', 'priority'])
  const { orderBy = 'created_at', order = 'DESC', limit, offset } = options

  const safeOrderBy = ALLOWED_ORDER_BY.has(orderBy) ? orderBy : 'created_at'
  const safeOrder   = order === 'ASC' ? 'ASC' : 'DESC'
  const safeLimit   = Number.isInteger(limit)  && limit  > 0 ? limit  : null
  const safeOffset  = Number.isInteger(offset) && offset >= 0 ? offset : null

  let sql = `SELECT * FROM user_stories WHERE status != 'archived' ORDER BY ${safeOrderBy} ${safeOrder}`
  if (safeLimit)  sql += ` LIMIT ${safeLimit}`
  if (safeOffset) sql += ` OFFSET ${safeOffset}`

  return db.prepare(sql).all()
}
```

## Prioritized Recommendations

### High Priority

1. **Escape input in `renderMarkdown()` before regex replacement** (Finding 1a): Add
   five `replace()` calls for `& < > " '` before the markdown rules. One function,
   fixes both call sites (lines 1882 and 4902) simultaneously.

### Medium Priority

2. **Escape `errorMessage` in `user-stories.js:1826`** (Finding 1c): Single-line fix —
   `this.escapeHtml(errorMessage)` or `escapeHtml` from any accessible scope.

3. **Add protocol allowlist to `shell:openExternal` and `webserver:openUrl`** (Finding 4a):
   Reject non-`http`/`https` URLs in both handlers. Blocks `file://`, `ftp://`, and
   Windows app-protocol abuse.

4. **Whitelist `marked` options in `markdown:parse`** (Finding 4b): Drop `renderer` and
   any undocumented keys from the options object. Prevents future option-injection if
   call sites change.

5. **Validate `findAll()` sort/page parameters** (Finding 6): Whitelist `orderBy`,
   coerce `order` to `'ASC'|'DESC'`, and integer-check `limit`/`offset`. Eliminates
   the latent SQL injection risk before any future IPC handler exposes these parameters.
